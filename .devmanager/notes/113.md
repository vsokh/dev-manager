# Task 113: Add ActionContext for App.tsx

## Manager notes
App.tsx destructures 36+ props and passes 42+ event handlers down. Create an ActionContext (or similar) so child components can consume actions directly instead of receiving them through layers of props.

## Exploration findings
- App.tsx currently passes 16+ props to TaskBoard, 16+ to CommandQueue, 11+ to TaskDetail
- Worst drilling: queue handlers go App → CommandQueue → PhaseView → QueueItemContent (3 levels)
- Task update handlers go through 2-3 levels to reach leaf components
- Selection state (selectedTask, glowTaskId) drills through TaskBoard → EpicGroup → TaskCard
- No React Context exists anywhere in the project yet
- Two main action hooks: useTaskActions (10 handlers) and useQueueActions (13 handlers + state)
- Also: pauseTask/cancelTask from useProject, selection state from App.tsx local state

## Proposed plan
1. Create `src/contexts/ActionContext.tsx` — a single context that bundles:
   - All task action handlers (from useTaskActions)
   - All queue action handlers (from useQueueActions)
   - pauseTask/cancelTask (from useProject)
   - Selection state (selectedTask, glowTaskId, handleSelectTask, handleNavigateToTask)
   - Config (defaultEngine)
2. Create `ActionProvider` component that wraps the app and provides all actions
3. Create `useActions()` hook as the consumer API
4. Update App.tsx to use ActionProvider
5. Update child components to use useActions() instead of receiving props:
   - TaskBoard.tsx — remove ~10 action props
   - CommandQueue.tsx — remove ~12 action props
   - TaskDetail.tsx — remove ~7 action props
   - PhaseView.tsx — remove ~10 action props
   - QueueItemContent.tsx — remove ~6 action props
   - EpicGroup.tsx — remove ~5 action props
   - ActionButtons.tsx — remove 3 action props
   - DoneSection.tsx — remove 2 action props
   - TaskCard.tsx — remove 2 action props
6. Update tests to wrap components in ActionProvider
